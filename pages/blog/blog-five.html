Using a Static Site Generator - Hugo

29/03/2020

Demonstrator - My Hugo website – pawelswies.com

It may seem like static site generators are for those who don’t code, and simply spit out a website for you to add your content to, and while this is one use case for them, static site generators also a powerful tool for developers creating sites from scratch.

Four years ago, Matt Biilmann (2015) predicted that static site generators are the next big thing, and he was right. Thanks to their features and speed, static generators have become increasingly popular. Especially now that online traffic has risen (Statista, 2020) due to the outbreak, it is important to keep sites fast. In a short blog post, Eric Meyer (2020) argues that sites that have important information on them should go static

“because too many sites are already crashing because their CMSes can’t keep up with the traffic surges”.

There are many static website generators to choose from; staticgen.com has a list of most of them, ranking them on popularity and providing more information about each one. I chose Hugo as it’s very well documented and since it’s one of the most popular site generators, there are many other resources and community forums, so learning about it and troubleshooting is very easy.

Installing Hugo

Getting started with Hugo is simple. On the website, there are instructions on multiple ways of installing it. Using Homebrew, I simply used the brew install hugo command and once the installation finished, I was ready to go. All the ways of installing Hugo, including the way I did it can be found on Hugo’s installing page.

Creating a New Hugo Website

Hugo is used purely in the command line. There are many commands that do many things, however I’ve only needed to use just a few commands which are easy to remember.

hugo new site example

This command will generate the boilerplate of a Hugo site with all default folders and config files. In this instance, it will all live in a folder called “example” as anything that comes after new site is the name of the folder the site will be generated in. This purely generates the folders and the config files, and no HTML or CSS is generated. Hugo supports themes, which are basically pre-made websites that can be downloaded into a new Hugo site and customised to the likes of the user. This can be useful to those who need a quick starter website or those who don’t know how to code. Today I will focus on building a Hugo site from scratch, but it’s worth mentioning that downloading a theme and seeing how it’s put together is a great way of seeing how Hugo works.

hugo new blog-post.md

This creates a new Markdown file called blog-post in the content folder. Files can be created manually, however a new file generated through Hugo is automatically generated with front matter at the top of the file, which includes meta data such as the date the file was created and the title of the page. The default front matter looks like this: 

---
title: "Blog Post"
date: 2020-03-29T21:05:42+01:00
draft: true
---

The front matter can be customised to include more information, such as tags, description and category to name a few. The full list of pre-defined front matter variables can be found on this page. The variables aren’t limited to the predefined ones; the user can also define their own to suit their needs.

The Markdown file is essentially what is displayed on the page, and the information about it, but this alone doesn’t create a web page because there’s no code. For the new page to exist and use the information in the Markdown file, a template HTML file needs to be created. In the layouts folder, a new folder called “_default” needs to be created, and inside of that a new file called single.html also needs to be created. This file is the template for all pages that are not the index pages.

Here is the single page template for my Hugo site:

{{ define "main" }}

{{ partial "header.html" . }}

<section class="post">
    <div class="post__wrap wrapper">
        <div class="post__heading">
            <h1 class="post__title">{{ .Title }}</h1> <!-- Grabs title from Markdown -->
            <p class="post__date">{{ .PublishDate.Format "January 2, 2006" }}</p> <!-- Grabs date from Markdown and prints it in "Month Day, Year" format -->
        </div>
        {{ .Content }} <!-- Grabs content from Markdown -->
    </div>
</section>

{{ end }}

This HTML will be generated for each new page. The {{ .Title }} grabs the title of the page from the Markdown file, and prints it on the site. The same goes for the date, however here I use a custom format setting that makes the date display in a specific format. There are multiple ways to format dates, some examples can be seen in Hugo’s documentation. To display the content from the Markdown file, {{ .Content }} has to be placed wherever you want the content to be. Without this, the content will simply not be displayed. Note that the {{ .Content }} tag isn’t wrapped in a semantic HTML tag. This is because those tags are automatically generated based on the MD file.

It’s important to note that new pages don’t have to use the single page template. Each page can use a custom layout template and content can be hardcoded onto the page rather than using content from the MD file.

Now, Hugo has a template to generate a page from and to place the content on to it from the Markdown file, however there is no base HTML page structure with the head, doctype etc. This is where the baseof.html file in the default layouts folder file comes in. This file is the template that wraps every page and contains the base HTML structure.

Here is an example from my site:

<!DOCTYPE html>
<html lang="en">
<head>
    <meta charset="UTF-8">
    <meta name="viewport" content="width=device-width, initial-scale=1.0">
    <meta name="description" content="{{ .Description }}"> <!-- Grabs description from Markdown -->
    <meta name="date" content="{{ .Date }}"> <!-- Grabs date from Markdown -->
    <title>{{ .Title }}</title> <!-- Grabs title from Markdown -->
    <link rel="stylesheet" href="/css/style.css">
    <link rel="stylesheet" href="/css/prism.css">
</head>
<body>

    {{ block "main" . }}

    {{ end }}

    {{ partial "footer.html" . }} <!-- Grabs footer partial -->

</body>
<script src="/js/prism.js"></script>
</html>

A baseof file is very useful as it contains a template for the head that works on all pages thanks to variables. Every time a new page is generated, the {{ }} variables pull the metadata from the MD file and automatically fill it in the head. You can also see that I include the footer partial inside the body, so that the footer shows on all pages.

hugo server

Hugo has live reloading built in, meaning that when this command in ran, Hugo locally hosts the website and refreshes it whenever a change is saved. There are many “flags” that can be added when running this command, but the only one I use is -D which makes draft pages also show up on the site. Draft pages have the draft setting set to true in their Markdown file. When the site is published, these draft pages don’t show up, however seeing them when working on the website is useful.

Finishing up

hugo

Before the site can be deployed onto a server, it needs to be rendered by Hugo so that a browser can display it. Simply using the hugo command will swiftly build the website into a “public” folder. Now, the contents of that folder can finally be deployed onto a server.
