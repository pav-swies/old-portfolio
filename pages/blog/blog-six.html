JAMstack Site with a CMS

06/04/2020

Demonstrator: jamstackforbeginners.netlify.com

As continuation of my Hugo website project, I looked into making it a JAMstack page and adding a content management to it. In this blog post I will explain what the JAMstack is, its benefits and how I set my site up to use it. I will also explain the process of implementing a static contemnt management to a static site. Along side of this project, I also created a website that helps beginners create their own JAMstack site.

What is the JAMstack?

The JAMstack is a differnet way of serving a website compared to another common stack, for example the LAMP stack (Potter, 2019). A common way of serving a website is by having it live on a server, and every time a user opens the site, a request is sent to the server. The server then grabs content from the database, then builds the site and sends it back to the user. In contrast, a JAMstack website does not rely on a webserver because it’s static. The website is pre-built, and is simply served to the user directly from a content delivery network.

Why is this better?

Since the sites are pre-rendered, the loading times are much faster compared to a traditional website on a server (Jamstack, no date). JAMstack removes the need for all of the moving parts a dynamic site requires, like a load balancer, web server, database etc. Not relying on these extra moving parts removes the possibility of a security breach on the webservers and databases, and makes hosting much cheaper (JAMstack WTF, no date). A static site simply needs to be served from a content delivery network. This also makes developing a JAMstack website a better experience. Services like Netlify even offer free hosting which is perfect for small or personal sites on a budget.

What about the name?

  • Javascript - Used for progressive interactivity. Since there is no server, the scripts aren’t ran on the server. The browser acts as the runtime.
  • APIs - Used for external dynamic data access.
  • Markup - Used for static content to be generated from.

A JAMstack website doesn’t need all of these to be a JAMstack site. Not all websites will have interactivity through JavaScript or external data passed into them through APIs.

How it works

Since the website is prerendered, it needs to be located somewhere where a CDN (Content Delivery Network) can access it. This is typically a Git repository.

For my JAMstack site, I uploaded the site to a GitHub repository, then imported it into Netlify. I used a static site generator called Hugo to make my site, and since Netlify works with Hugo by default, it’s very easy to get it running.

Static Website Generators

Static site generators create their own source files then convert/build them into a static website. SSG’s have many very useful features that allow for a more powerful site. There are many static site generators to choose from; staticgen.com is a great website with a huge list of SSG’s you could use. For my personal JAMstack site, I used Hugo and found it to be really good to work with. You can read my blog about using Hugo here.

Again, you do not have to use a static site generator for it to be a static website serverd by a CDN. It’s also important to note that just because it’s a “static” website, it does not mean that the experience has to be static. Using JavaScript there are many things that can be done to create a dynamic experience.

Using Netlify CDN

To host a website on Netlify, simply make an account, then sign in to GitHub and select the repository and branch your site is in. All that’s left to do is to give it a name, and it’s ready. Netlify will automatically give you a domain and your site will now be live.

Automated builds

Every time a change is uploaded to the GitHub repo, Netlify will automatically run the Hugo build task and deploy the website. The build task only needs to run once, then the site is ready to be served.

Content Management Systems

Now, in my opinion this is the best part of this whole setup. While not exactly part of the JAMstack, static content management systems are able to add content to your website without having to touch any code. It’s a a great way to add tons of functionality to you website without having to do much.

How it works

Opposed to a server-side CMS like WordPress, a static CMS is not on a server. For my JAMstack site, I used Forestry CMS because it natively works with Hugo. All that I had to do to integrate it into my website is create a free account and import the GitHub repository of my website.

Forestry provides a sleek user-interface where you can see and edit the content (Markdown files) of the website. Whenever a change is saved in Forestry, it pushes those changes to the GitHub repository of the site. Now, because the site is connected to Netlify, the automated build is ran and the website is updated within seconds. It’s extremely easy and useful and I would recommend this to anyone using a static site generator. There are many more static CMS’s, headlesscms.org has a large list of them.
